Byzantine Generals Problem

Introduction

blockchain是一種分散式資料庫架構，此系統中，多個Node以”非同步”方式組成blockchain網路，在這樣的一個非同步系統中，需要Node彼此之間進行state duplication，以確保一致的狀態。

但是非同步系統中，主機有可能故障、網路可能壅塞，這些因素會導致錯誤訊息在系統內傳播，因此需要設計一個容錯技術以確保主機之間可安全且正確通訊。

Byzantine Generals Problem

Leslie Lamport在20世紀80年代提出一個假想問題：

Byzantine是東羅馬帝國首都，當時幅員遼闊，因此”將軍”們彼此之間傳遞消息是一門學問，一旦出現叛徒將軍，將有可能會影響作戰計畫的進行。

所以，Byzantine Generals Problem指的是：”找一個方法，使得將軍們能夠在一個有叛徒將軍的truthless環境中，建立作戰計畫的共識”

所以就這麼剛好，Byzantine就跟Blockchain的分散式系統問題相類似。

  • 忠誠將軍-> 正常Node
  • 叛變將軍-> 損壞的Node

Byzantine Generals Problem可以描述為：

  • 所有忠誠的且接收到命令的將軍遵守相同的命令
  • 如果發送命令的將軍是忠誠的，那麼其他忠誠的將軍遵守其發出的命令

Lamport對Byzantine Generals Probloem問題的研究指出：
當叛徒個數小於將軍總數的1/3時，透過口頭”同步”通訊，可以滿足以上兩個問題。
而如果在通訊是可認證、防篡改情況底下(PKI、數位簽章等)，則在任意多的叛徒的情況底下可以找到解決方案。

但是，非同步通訊就無法。

Consensus Problem

Consensus Problem就是找出一個協定或算法，其必須滿足：Agreement、Validity、Termination。

Fischer-Lynch-Paterson理論指出，在非同步通訊的分散式系統當中，只要有一個Byzantine Fault，就不可找到一個演算法，同時滿足Agreement(正常Node必須同意一個值)、Validity(正常Node同意同一個初始值)、Termination(正常Node必須最終確定一個值)。

專有名詞介紹：
Byzantine Fault：任何觀察者從不同角度看，會表現出不同症狀的缺陷。
Byzantine Failure：因Byzantine Fault而喪失服務。

在private chain和consortium chain中，對Agreement要求相當高
在public chain中，通常採用Eventual Consistency共識算法
